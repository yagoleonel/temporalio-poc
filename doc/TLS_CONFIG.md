# TLS Configuration

This document describes the mTLS (mutual TLS) configuration used in this Temporal cluster,
based on the official [`tls-simple`](https://github.com/temporalio/samples-server/tree/main/tls/tls-simple) sample
from `temporalio/samples-server`.

## Overview

mTLS is enabled entirely via environment variables. The Temporal Docker image ships a
[`config_template.yaml`](https://github.com/temporalio/docker-builds/blob/main/docker/config_template.yaml)
(a [dockerize](https://github.com/jwilder/dockerize) Go template) that generates the server
configuration at container startup. The `TEMPORAL_TLS_*` env vars are substituted into this
template to produce the final `global.tls` configuration block.

There are **two TLS scopes** in Temporal:

| Scope | Purpose |
|---|---|
| **Internode** | Secures communication between cluster services (history, matching, frontend, worker) |
| **Frontend** | Secures the gRPC endpoint exposed to external clients (SDKs, CLI, UI) |

Each scope has a **server** side (presents a certificate) and a **client** side (verifies the
remote certificate).

## Certificates

Certificates are generated by `generate-test-certs.sh` and stored in `certs/`.

| File | Purpose | Used by |
|---|---|---|
| `ca.cert` / `ca.key` | Certificate Authority | Signs all certs |
| `cluster.pem` / `cluster.key` | Cluster certificate (serverAuth + clientAuth) | All Temporal server services |
| `client.pem` / `client.key` | Client certificate (clientAuth only) | External clients (UI, SDKs, CLI) |

### Certificate configs

- **`cluster-cert.conf`** — OpenSSL config for the cluster certificate. Contains DNS SANs for all
  service hostnames (`temporal-poc`, `temporal-history`, `temporal-matching`, `temporal-frontend`,
  `temporal-worker`, `localhost`, `internode`) and `IP.1 = ::1`.

- **`client-cert.conf`** — OpenSSL config for the client certificate. Contains DNS SANs for
  `localhost` and IP SANs for `::1` and `127.0.0.1`.

## Environment Variables

All env vars below are set on every Temporal server service (`temporal-history`, `temporal-matching`,
`temporal-frontend`, `temporal-worker`) in `docker-compose.yml`.

### Internode TLS — Server Side

Configures the TLS certificate that each Temporal service presents to other services within
the cluster.

Maps to: `global.tls.internode.server` in `config_template.yaml`.

| Variable | Value | Description |
|---|---|---|
| `TEMPORAL_TLS_SERVER_CA_CERT` | `/etc/temporal/config/certs/ca.cert` | CA certificate used to verify internode client certificates (`clientCaFiles`) |
| `TEMPORAL_TLS_SERVER_CERT` | `/etc/temporal/config/certs/cluster.pem` | TLS certificate presented by internode servers (`certFile`) |
| `TEMPORAL_TLS_SERVER_KEY` | `/etc/temporal/config/certs/cluster.key` | Private key for the internode server certificate (`keyFile`) |
| `TEMPORAL_TLS_REQUIRE_CLIENT_AUTH` | `true` | Require connecting clients to present a certificate — enables mTLS (`requireClientAuth`). Applies to **both** internode and frontend scopes |

### Frontend TLS — Server Side

Configures the TLS certificate that the Frontend service presents to external clients
(SDKs, CLI, UI).

Maps to: `global.tls.frontend.server` in `config_template.yaml`.

| Variable | Value | Description |
|---|---|---|
| `TEMPORAL_TLS_CLIENT1_CA_CERT` | `/etc/temporal/config/certs/ca.cert` | CA certificate used to verify external client certificates (`clientCaFiles[0]`). The template supports up to two client CAs (`CLIENT1` and `CLIENT2`) for scenarios where internode and external clients use different CAs. The `1` in the name is part of the variable name, not a counter |
| `TEMPORAL_TLS_FRONTEND_CERT` | `/etc/temporal/config/certs/cluster.pem` | TLS certificate presented by the Frontend server to external clients (`certFile`) |
| `TEMPORAL_TLS_FRONTEND_KEY` | `/etc/temporal/config/certs/cluster.key` | Private key for the Frontend server certificate (`keyFile`) |

### Internode TLS — Client Side

Configures how Temporal cluster services connect to other internode services (history,
matching, internal-frontend) over TLS.

Maps to: `global.tls.internode.client` in `config_template.yaml`.

| Variable | Value | Description |
|---|---|---|
| `TEMPORAL_TLS_INTERNODE_SERVER_NAME` | `temporal-poc` | Expected server name in internode certificates, used for TLS hostname verification (`serverName`). Must match a SAN in the cluster certificate |
| `TEMPORAL_TLS_INTERNODE_DISABLE_HOST_VERIFICATION` | `false` | When `false`, verifies the internode server's hostname against its certificate (`disableHostVerification`) |

> The internode client uses `TEMPORAL_TLS_SERVER_CA_CERT` as its `rootCaFiles` to verify the
> remote server's certificate.

### Frontend TLS — Client Side

Configures how Temporal cluster services (specifically the Worker role) connect to the
Frontend service over TLS.

Maps to: `global.tls.frontend.client` in `config_template.yaml`.

| Variable | Value | Description |
|---|---|---|
| `TEMPORAL_TLS_FRONTEND_SERVER_NAME` | `temporal-poc` | Expected server name in the Frontend certificate, used for TLS hostname verification (`serverName`). Must match a SAN in the cluster certificate |
| `TEMPORAL_TLS_FRONTEND_DISABLE_HOST_VERIFICATION` | `false` | When `false`, verifies the Frontend server's hostname against its certificate (`disableHostVerification`) |

> The frontend client also uses `TEMPORAL_TLS_SERVER_CA_CERT` as its `rootCaFiles` to verify
> the Frontend server's certificate.

### Temporal CLI TLS (auto-setup only)

These variables are only set on `temporal-history` (which uses the `temporalio/auto-setup` image).
The auto-setup script runs `temporal operator cluster health` to wait for the Frontend to become
available, then registers the default namespace. Since the Frontend requires mTLS, the CLI
needs these credentials.

| Variable | Value | Description |
|---|---|---|
| `TEMPORAL_TLS_CA` | `/etc/temporal/config/certs/ca.cert` | CA certificate for the CLI to verify the Frontend server (`--tls-ca-path`) |
| `TEMPORAL_TLS_CERT` | `/etc/temporal/config/certs/cluster.pem` | Client certificate for the CLI to present to the Frontend (`--tls-cert-path`) |
| `TEMPORAL_TLS_KEY` | `/etc/temporal/config/certs/cluster.key` | Client private key for the CLI (`--tls-key-path`) |
| `TEMPORAL_TLS_ENABLE_HOST_VERIFICATION` | `true` | Verify the Frontend server's hostname (`--tls-enable-host-verification`) |
| `TEMPORAL_TLS_SERVER_NAME` | `temporal-poc` | Override the TLS server name for verification (`--tls-server-name`) |

### Temporal UI TLS

These variables are set on `temporal-ui` to connect to the Frontend over mTLS.

| Variable | Value | Description |
|---|---|---|
| `TEMPORAL_TLS_CA` | `/etc/temporal/config/certs/ca.cert` | CA certificate to verify the Frontend server |
| `TEMPORAL_TLS_CERT` | `/etc/temporal/config/certs/client.pem` | Client certificate presented to the Frontend |
| `TEMPORAL_TLS_KEY` | `/etc/temporal/config/certs/client.key` | Client private key |
| `TEMPORAL_TLS_ENABLE_HOST_VERIFICATION` | `true` | Verify the Frontend server's hostname |
| `TEMPORAL_TLS_SERVER_NAME` | `temporal-poc` | Override the TLS server name for verification |

> The UI uses the **client** certificate (`client.pem`), not the cluster certificate.

## Volume Mounts

All services mount certificates at the same path:

```yaml
volumes:
  - ./certs:/etc/temporal/config/certs
```

This path (`/etc/temporal/config/certs`) follows the convention from the `tls-simple` sample.

## Connecting to the Cluster

### With grpcurl

```bash
grpcurl \
  -cacert ./certs/ca.cert \
  -cert ./certs/client.pem \
  -key ./certs/client.key \
  -servername temporal-poc \
  localhost:7233 \
  temporal.api.workflowservice.v1.WorkflowService/GetSystemInfo
```

### With the Temporal CLI

```bash
temporal operator namespace list \
  --tls-ca-path ./certs/ca.cert \
  --tls-cert-path ./certs/client.pem \
  --tls-key-path ./certs/client.key \
  --tls-server-name temporal-poc
```

### With the Go SDK

```go
caPEM, _ := os.ReadFile("certs/ca.cert")
rootPool := x509.NewCertPool()
rootPool.AppendCertsFromPEM(caPEM)

clientCert, _ := tls.LoadX509KeyPair("certs/client.pem", "certs/client.key")

c, _ := client.Dial(client.Options{
    ConnectionOptions: client.ConnectionOptions{
        TLS: &tls.Config{
            RootCAs:      rootPool,
            Certificates: []tls.Certificate{clientCert},
            ServerName:   "temporal-poc",
        },
    },
})
```

## Disabling Client Authentication

To use TLS without mTLS (server-only TLS, no client certificate required), set
`TEMPORAL_TLS_REQUIRE_CLIENT_AUTH=false` on all server services and restart the cluster.
Clients will still need to trust the CA but won't need to present their own certificate.
